# AI Workstation: Workflows & Collaboration Guidelines

This document details the operational rules for AI assistants (like Gemini / Antigravity CLI) working on this codebase. It ensures high code quality, security compliance, clean package boundaries, and smooth multi-agent cooperation.

---

## 1. Planning Workflow
Before making any file modifications or proposing structural changes, the AI must follow this rigorous 11-step process:

1. **Read `GEMINI.md`**: Understand the root rules and current workstation baseline.
2. **Read `docs/ai/context.md`**: Align with product goals and runtime limits.
3. **Read `docs/ai/architecture.md`**: Review package boundaries and layering constraints.
4. **Read `docs/ai/roadmap.md`**: Ensure the task aligns with the roadmap boundaries.
5. **Read `docs/ai/tasks.md`**: Check active status and verify parent task IDs.
6. **Identify Current Phase**: Pinpoint the precise development scope.
7. **Propose Implementation Plan**: Draft a localized, highly specific plan for the user's approval.
8. **Target Modifications**: Identify only the specific files that need changing.
9. **Formulate Test Plan**: Draft the required unit and integration tests.
10. **Execute Validation Checklist**: Plan compile and execution assertions.
11. **Update Active Task Status**: Outline the changes to apply to `docs/ai/tasks.md`.

---

## 2. Implementation Workflow
- **Keep Changes Localized**: Do not initiate large-scale codebase refactorings unless explicitly agreed upon in a design discussion.
- **Implement One Slice at a Time**: Work on small, vertical slices (e.g., entity -> repository -> service -> controller -> test) instead of writing multiple independent layers simultaneously.
- **Thin Controllers**: Controllers must only process request inputs, delegate processing to services, and return validated DTOs.
- **Service Isolation**: Put all business logic, prompt merges, and validation logic into services.
- **Versioned & Decoupled Prompts**: Do not hardcode raw prompt strings in services. Manage prompts using external resources or distinct template managers.
- **Pre-return JSON Validation**: Every response generated by local LLMs must pass parsing and validation rules before being returned by the service layer.
- **No Premature Abstractions**: Avoid generic wrappers or helper classes for future modules (such as Social Network Analysis or security scanning).

---

## 3. Testing Workflow
- **Service-Level Unit Tests**: Mock out databases and the Spring AI `ChatModel` in service unit tests. These must run instantly during normal Maven build cycles without external requirements.
- **Controller-Level Integration Tests**: Use `MockMvc` and `@WebMvcTest` to test controllers, verifying status codes, JSON mappings, and input validation bounds.
- **Optional Live AI Integration Tests**: Live Ollama tests must be marked integration-only and kept optional. They should run only when a specific Spring profile (`integration-test`) is active.
- **Standardized Fixtures**: Store reusable test CVs, job ads, and expected JSON structures in `src/test/resources/fixtures/` rather than hardcoding them inside test classes.

---

## 4. Validation Workflow
Before declaring a task "Done", compile and execute:
1. **Compile Verification**: Execute `mvn clean compile` to ensure there are no compilation errors or import warnings.
2. **Test Executions**: Run `mvn test` to confirm the entire test suite passes without errors.
3. **JSON Schema Audits**: Verify that structured API payloads comply with OpenAPI/Swagger structures.
4. **Security Implications Checklist**: Ensure all file inputs are sanitised and no private user details are exposed to terminal outputs.
5. **Deployment Compatibility**: Ensure that no absolute file paths or local-only configurations are hardcoded into application properties.
6. **Documentation Audit**: If design choices were modified, ensure that `docs/ai/architecture.md` and related files are kept aligned.

---

## 5. Multiple Sub-Agent Collaboration Workflow
When complex parallel tasks require dividing labor among multiple AI agents, we enforce a highly structured multi-agent workflow.

### Dedicated Collaboration Roles
- **Architect Agent**: Defines module boundaries, reviews class packaging, and audits ADR compliance.
- **Backend Agent**: Implements the Spring Boot application logic, controllers, services, and repositories.
- **AI/RAG Agent**: Implements prompt structures, semantic chunking pipelines, embedding calculations, and PGVector operations.
- **Security Agent**: Focuses exclusively on file ingestion sanitization, prompt injection prevention, data minimization, and PII log scrubbing.
- **Test Agent**: Focuses on writing mock fixtures, unit assertions, controller contract tests, and integration suites.
- **Web Product Agent**: Formulates endpoints, handles standard browser integration structures, and maps CORS/security origins.
- **Deployment Agent**: Reviews environmental variable mappings, Spring profiles, and production secrets storage files.
- **Review Agent**: Performs the final quality gate check, evaluating code against styleguides, roadmap boundaries, and architectural patterns.

### Concurrent Implementation Guidelines
- **Zero-Conflict Boundaries**: No two sub-agents should edit the exact same file concurrently.
- **Shared File Serialization**: Updates to shared resources (e.g. `pom.xml`, `application.properties`, or `tasks.md`) must be executed sequentially.
- **Task Constraints Matrix**: Every parallel task assigned to an agent must define:
  - **Owner Agent**: The specific sub-agent ID.
  - **Allowed Files**: A whitelist of absolute file paths the agent is permitted to write or edit.
  - **Forbidden Files**: A blacklist of files the agent must not modify under any circumstance.
  - **Success Criteria**: Clear, binary execution criteria.

### Strict Merge Sequence Order
When combining parallel feature branches or merging code slices, changes must be processed in this exact sequence to prevent dependency breakage:
1. **Architecture & ADRs**: Confirm high-level design structures are written and approved.
2. **Dependencies & System Configuration**: Merge core library imports (like `pom.xml` changes) and config beans.
3. **Domain Models & DTO Contracts**: Merge entities and API transfer contracts.
4. **Services & Utilities**: Merge business logic services and AI pipeline classes.
5. **Controllers & Endpoints**: Merge HTTP controllers and REST configurations.
6. **Tests**: Merge the verification suites.
7. **Documentation & Task Board**: Update task states in `tasks.md` and context files.
