# AI Workstation: Factual Validation & Guardrails

This document defines the validation procedures, validation rules, and non-negotiable factual guardrails to ensure the system is secure, factual, and structurally correct.

---

## Non-Negotiable AI Guardrails

### Factual Integrity Rules
All runtime AI components (including prompts, validators, and parser services) must enforce absolute truth. To protect the credibility of our users and our application, the system **MUST NEVER**:
1. **Invent Work Experience**: Do not synthesize hypothetical roles, company names, or work descriptions that are not in the raw document text.
2. **Fabricate Education**: Never add degrees, institutions, or educational backgrounds not explicitly listed by the user.
3. **Manufacture Certifications**: Do not suggest or append certifications (e.g. AWS, Scrum Master, PMP) that the user does not possess.
4. **Invent Skills**: Do not inject technical or soft skills that the user has not demonstrated or stated in their parsed history.
5. **Alter Chronological Facts**: Never change employment dates, graduation periods, or duration values unless a user explicitly requests an adjustment to correct a typo.
6. **Alter Identity Information**: Names, email addresses, phone numbers, and portfolio links must remain exactly as originally parsed unless explicitly modified by the user.
7. **Overclaim Job Alignment**: Never claim a CV matches a job requirement if there is no documented evidence in the CV to back it up.
8. **Assert Hiring Guarantees**: AI feedback must never present suggestions as "guarantees" of employment, hiring success, or specific salary brackets.

## Validation Protocols & Acceptance Criteria

### 1. Document Upload & Ingestion Validation
Before accepting any byte array, the `DocumentController` must enforce:
- **File Ingestion Size Limit**: Reject any file exceeding **5MB**.
- **Mime Type Magic Number Scan**: Read file header magic numbers to ensure the file is a true PDF (`%PDF-`) or DOCX (`PK\x03\x04`). Reject files matching only name extensions.
- **Empty Stream Check**: Ensure the extracted text length exceeds at least **100 characters** to prevent blank document errors.

### 2. Structured JSON Output Validation
Since local LLMs (`gemma4:2eb`) may deviate from formatted schema templates, all analysis services must implement a validation wrapper:
- **JSON Schema Verification**: The raw text output must be parsed using Jackson `ObjectMapper`. If it is not valid JSON, trigger a secondary prompt retry or fallback parser.
- **Null & Boundary Scans**: Confirm required fields (e.g., `atsScore`, `strengths`, `weaknesses`) are non-null and that scores lie between **0 and 100**.

### 3. Factual Integrity Auditing (Rewrites)
Suggested optimizations generated by the LLM rewrite engine must pass an automated comparison audit:
- **Noun & Metric Verification**: Suggested rewrites are scanned for numeric metrics (e.g. "managed $50k budget"). If a numeric metric or tool name is added that did not exist in the original CV sentence or target keyword list, the suggestion is immediately flagged as a hallucination.
- **Factual Mapping Check**: Check that all company names and dates are 1:1 identical to the original block.

### 4. RAG Retrieval Validation
When pulling CV segments based on job ad similarity:
- **Min Relevance Threshold**: Set a minimum similarity score threshold (e.g., **0.75 cosine similarity**). Chunks below this threshold must not be sent to the LLM to prevent irrelevant context.

## Detailed Component Acceptance Criteria

### CV Analysis Response
The structured JSON payload returned by POST `/api/cv/analyze` must conform to:
```json
{
  "atsScore": 75,
  "strengths": [
    "Clear, action-oriented action verbs in professional experience section",
    "Detailed technical skills list matching standard industry taxonomy"
  ],
  "weaknesses": [
    "Lack of quantifiable metrics or direct project achievements",
    "Missing clear summary statement at the top"
  ],
  "recommendations": [
    "Quantify your work: add metrics like budget sizes, user growth, or performance optimizations.",
    "Add a brief, highly targeted professional profile summarizing key skills."
  ]
}
```

### Job Matching Response
The structured JSON payload returned by POST `/api/jobs/match` must conform to:
```json
{
  "matchPercentage": 65,
  "missingKeywords": ["Docker", "CI/CD Pipelines", "JUnit 5"],
  "matchingStrength": "Strong overlap in core Java programming and Spring Boot development.",
  "actionableSuggestions": [
    "Incorporate your JUnit testing experience in the resume highlights section to match the testing requirement."
  ]
}
```

### Rewrite Response
The structured JSON payload returned by POST `/api/rewrite/section` must conform to:
```json
{
  "originalText": "Responsible for managing the Java database connections.",
  "suggestions": [
    {
      "style": "ACTION_ORIENTED",
      "text": "Engineered and optimized Java database connections to maximize data access speeds."
    },
    {
      "style": "METRIC_DRIVEN",
      "text": "Managed Java database connections, maintaining system uptime across production systems."
    },
    {
      "style": "CONCISE",
      "text": "Managed Java database connection optimization."
    }
  ]
}
```

## Human-in-the-Loop Requirements
- **Interactive Confirms**: The application must never directly apply or mutate a user's original document. It must display AI suggestions visually, requiring the user to explicitly click "Accept" or "Apply" to execute changes.
- **Factual Disclaimer**: All AI generated optimization screens must display a visible notice: *"Please review suggestions carefully to ensure they match your authentic professional history."*
